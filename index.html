La Fundamentación del Diseño

El diseño de bases de datos es una colección de procesos sistemáticos que facilitan el desarrollo, implementación y mantenimiento de los datos empresariales.

Entidades y Atributos

Abstracción de objetos del mundo real (Entidades) y sus rasgos característicos (Atributos) para definir la estructura base del sistema.

Cardinalidad

Definición de las restricciones de participación (1:1, 1:N, N:M) entre entidades para reflejar las reglas de negocio.

Integridad Referencial

Uso de claves primarias (PK) y foráneas (FK) para asegurar la consistencia y coherencia de los datos en toda la arquitectura.

Normalización

Proceso de refinamiento de tablas para eliminar redundancias y evitar anomalías en las operaciones de inserción, actualización y borrado.

Optimización Estructural

Las bases de datos diseñadas correctamente son fáciles de mantener, mejoran la consistencia y son rentables en términos de almacenamiento físico.

Fase 01

Modelado Conceptual: Caso Concesionario

Abstracción del flujo de datos del mundo real para la **Gestión Integral de Concesionarios y Servicios Posventa**.

Diagrama Entidad-Relación (E-R)

Análogo de Reglas de Negocio

  • Identificación Unívoca: Cada concesionario se identifica por su NIT, permitiendo la gestión descentralizada de múltiples sedes.
  • Gestión de Empleados: Los empleados están vinculados obligatoriamente a un concesionario, centralizando la nómina y el seguimiento de objetivos.
  • Trazabilidad de Ventas: Cada venta asocia un empleado (responsable) y un auto (producto), estableciendo una relación ternaria implícita de alta granularidad.
  • Entidades Débiles: El **Servicio Posventa** depende existencialmente del Auto y del historial de mantenimiento, requiriendo una identificación compuesta.
Fase 02

Diseño Lógico y Diccionario de Datos

Transformación del modelo semántico a esquemas relacionales normalizados y optimizados.

Catálogo de Entidades Técnicas

Tabla Clave Primaria Atributos Críticos Función Relacional
Concesionario Nit (CHAR) Nombre, Dirección, Teléfono. Nodo raíz del sistema.
Auto Código (CHAR) Marca, Modelo, Cilindraje, Precio. Activo transaccional principal.
Empleado Identificación (CHAR) Nombre, Fecha_Vinculación. Actor operativo y de ventas.
Venta Código_Venta (CHAR) Fecha, Valor_Pagado, Placa. Módulo de ingresos y auditoría.
ServicioPosventa Código_Servicio (CHAR) Placa, Fecha, Estado, Novedades. Gestión de garantías y reparaciones.

Lógica de Mapeo y Normalización

Manejo de Relaciones M:N

La relación entre **ServicioPosventa** y **Repuesto** se resolvió mediante la tabla asociativa **Servicio_Repuesto**, que captura la *Cantidad* utilizada, garantizando la escalabilidad del inventario.

Integridad de Dominio

Se definieron tipos de datos específicos: `VARCHAR` para descripciones flexibles y `CHAR` para identificadores de longitud fija, optimizando el rendimiento de los índices.

Normalización Avanzada (1NF a 3NF)

Se aplicó el rigor de Normalización para eliminar redundancias. Por ejemplo, en la tabla **Auto**, se separaron los atributos del **Concesionario** para evitar dependencias transitivas (3NF).

Claves Primarias y Candidatas

Identificación de conjuntos mínimos de atributos que garantizan la unicidad del registro sin ambigüedad técnica.

Restricciones Referenciales

Implementación de cláusulas ON DELETE RESTRICT para proteger la integridad histórica de los datos transaccionales.

Fase 03

Implementación Física: Ingeniería SQL

Codificación DDL/DML para el despliegue del motor de base de datos relacional.

Data Definition (DDL) 

CREATE TABLE Concesionario (
    Nit CHAR(20) PRIMARY KEY,
    Nombre VARCHAR(100) NOT NULL,
    Direccion VARCHAR(200) NOT NULL,
    Telefono CHAR(15) NOT NULL
);

CREATE TABLE Auto (
    Codigo CHAR(20) PRIMARY KEY,
    Marca VARCHAR(50) NOT NULL,
    Modelo VARCHAR(50) NOT NULL,
    Nit_Concesionario CHAR(20),
    FOREIGN KEY (Nit_Concesionario) 
        REFERENCES Concesionario(Nit)
);

CREATE TABLE Empleado (
    Identificacion CHAR(20) PRIMARY KEY,
    Nombre VARCHAR(100) NOT NULL,
    Nit_Concesionario CHAR(20),
    FOREIGN KEY (Nit_Concesionario) 
        REFERENCES Concesionario(Nit)
);
Analítica de Datos (Queries) 

-- Consulta de Ventas de Alto Valor
-- Cruce de 3 tablas con filtros de negocio
SELECT v.Codigo_Venta, a.Marca, e.Nombre
FROM Venta v
JOIN Auto a ON v.Codigo_Auto = a.Codigo
JOIN Empleado e ON v.Identificacion_Empleado = e.Identificacion
WHERE v.Valor_Pagado > 20000;

-- Análisis de Repuestos por Marca
SELECT p.Nombre, pm.Marca
FROM Proveedor p
JOIN Proveedor_Marca pm 
    ON p.ID_Proveedor = pm.ID_Proveedor;

Resumen de la Implementación

El sistema soporta el ciclo completo de negocio: desde la adquisición de inventario (**Proveedores**) hasta la fidelización (**Servicio Posventa**). La mayor complejidad radicó en el manejo de **restricciones de integridad referencial**, asegurando que no se eliminen registros maestros (Concesionarios) con dependencias activas (Vendedores/Autos).
